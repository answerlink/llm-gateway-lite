# LLM Gateway 路由策略说明

## 概述

LLM Gateway 支持灵活的 Provider 选择策略，通过 `default_provider` 和 `weight` 两个配置项实现**优先级路由**和**负载均衡**。

## 配置项说明

### 1. `default_provider`（模型级别）

**配置位置**: `models.yaml` 中的 `policy.default_provider`

**作用**: 指定该模型的首选 Provider

**优先级**: 🔴 **最高**（优先于 weight）

```yaml
# models.yaml
models:
  glm-4.7-flash:
    provider_map:
      glm-official: "GLM-4.7-FlashX2"
      openrouter: "z-ai/glm-4.7-flash"
    policy:
      default_provider: "glm-official"  # 优先使用官方 API
```

### 2. `weight`（Provider 级别）

**配置位置**: `providers.yaml` 中每个 provider 的 `weight`

**作用**: 当 default_provider 不可用时，按权重分配流量

**优先级**: 🟡 **中等**（兜底策略）

**默认值**: `1`

```yaml
# providers.yaml
providers:
  glm-official:
    weight: 5  # 权重 5 = 71% 概率 (5/(5+2))
  openrouter:
    weight: 2  # 权重 2 = 29% 概率 (2/(5+2))
```

## 路由选择逻辑

```
请求到达
  ↓
步骤1: 检查 default_provider
  ├─ 已配置 && 可用 → ✅ 直接使用
  └─ 未配置 || 不可用 → 进入步骤2
      ↓
步骤2: 按 weight 加权随机选择
  ├─ 收集所有可用的 provider
  ├─ 按权重计算概率
  └─ ✅ 返回选中的 provider
```

### Provider 不可用的情况

- 没有可用的 API Key（所有 key 都在 cooldown）
- 在重试时被 `exclude_providers` 排除
- 不在模型的 `allow_providers` 白名单中

## 典型应用场景

### 场景一：主备模式（推荐）

**适用于**: 有明确的主力 Provider 和备用 Provider

**特点**: 
- 优先使用成本更低/延迟更低的 Provider
- 主力不可用时自动切换到备用
- 适合生产环境的高可用部署

**配置示例**:

```yaml
# models.yaml
models:
  glm-4.7-flash:
    provider_map:
      glm-official: "GLM-4.7-FlashX2"    # 官方 API（便宜、快）
      openrouter: "z-ai/glm-4.7-flash"   # 第三方（贵、备用）
    policy:
      allow_providers:
        - glm-official
        - openrouter
      default_provider: "glm-official"   # 优先官方

# providers.yaml
providers:
  glm-official:
    base_url: "https://open.bigmodel.cn"
    keys:
      - id: "key-1"
        value: "xxx"
    weight: 10  # 兜底时也优先选择
    
  openrouter:
    base_url: "https://openrouter.ai"
    keys:
      - id: "key-1"
        value: "xxx"
    weight: 1   # 真正的最后备份
```

**流量分配**:
```
正常情况:
  glm-official: 100%
  openrouter:   0%

glm-official 故障时:
  glm-official: 0%
  openrouter:   100%
```

### 场景二：负载均衡模式

**适用于**: 多个 Provider 成本/性能相近，需要分散流量

**特点**:
- 不设置 default_provider
- 完全按 weight 比例分配流量
- 适合防止单点过载、降低限流风险

**配置示例**:

```yaml
# models.yaml
models:
  gpt-4o:
    provider_map:
      openai: "gpt-4o"
      azure-east: "gpt-4o"
      azure-west: "gpt-4o"
    policy:
      allow_providers:
        - openai
        - azure-east
        - azure-west
      # 不配置 default_provider，完全按权重分配

# providers.yaml
providers:
  openai:
    base_url: "https://api.openai.com"
    keys:
      - id: "key-1"
        value: "xxx"
    weight: 5   # 50% 流量
    
  azure-east:
    base_url: "https://east.openai.azure.com"
    keys:
      - id: "key-1"
        value: "xxx"
    weight: 3   # 30% 流量
    
  azure-west:
    base_url: "https://west.openai.azure.com"
    keys:
      - id: "key-1"
        value: "xxx"
    weight: 2   # 20% 流量
```

**流量分配**:
```
正常情况:
  openai:      50% (5/10)
  azure-east:  30% (3/10)
  azure-west:  20% (2/10)

openai 故障时:
  openai:      0%
  azure-east:  60% (3/5)
  azure-west:  40% (2/5)
```

## 重试场景下的行为

当某个 Provider 返回错误需要重试时：

1. **排除故障 Provider**: 将其加入 `exclude_providers`
2. **重新路由**: 即使它是 default_provider 也会被跳过
3. **选择备用**: 从剩余的 provider 中按 weight 选择

**示例**:

```yaml
# 配置
models:
  glm-4.7-flash:
    policy:
      default_provider: "glm-official"
providers:
  glm-official:
    weight: 5
  openrouter:
    weight: 2
```

```
第一次请求:
  → 使用 glm-official (default_provider)
  → 返回 400 错误（模型不存在）

重试请求:
  → 排除 glm-official
  → 只剩 openrouter
  → 使用 openrouter
  → 成功 ✅
```

## 最佳实践

### 1. 生产环境推荐配置

```yaml
# 主备模式 + 合理的权重
models:
  my-model:
    policy:
      default_provider: "primary"      # 主力
      
providers:
  primary:
    weight: 10    # 主力 provider
  backup-1:
    weight: 5     # 第一备份
  backup-2:
    weight: 1     # 最后备份
```

### 2. 多 Key 配合使用

```yaml
providers:
  openai:
    keys:
      - id: "key-1"
        value: "xxx"
      - id: "key-2"
        value: "yyy"
      - id: "key-3"
        value: "zzz"
    weight: 5
```

- 同一个 provider 的多个 key 会自动轮询
- 某个 key 限流时会自动 cooldown 并换下一个
- 所有 key 都不可用时才会切换 provider

### 3. 成本优化策略

```yaml
# 优先使用便宜的，贵的作为备份
models:
  my-model:
    policy:
      default_provider: "cheap-provider"  # 便宜的优先
      
providers:
  cheap-provider:
    weight: 10
  expensive-provider:
    weight: 1  # 仅作为紧急备份
```

## 常见问题

### Q: weight 为 0 会怎样？

A: weight 会被转换为 `tonumber(provider.weight) or 1`，0 会被当作 0 处理，该 provider 不会被选中（除非它是 default_provider）。

### Q: 所有 provider 都不可用会怎样？

A: 返回 503 错误，错误信息为 `no provider available`。

### Q: 能否动态调整 weight？

A: 可以，修改配置文件后会自动重载（默认 5 秒检查一次）。

### Q: default_provider 故障后会自动恢复吗？

A: 会。如果只是临时故障（如 5xx 错误），下次请求会再次尝试 default_provider。如果是 key cooldown，等待 cooldown 时间过后会自动恢复。
